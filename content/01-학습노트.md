# 1부. 이론 — 무엇을 공부해야 하나

## 1-1. 인증제도 개요

- **ISMS-P** = 정보보호(ISMS) + 개인정보보호(PIMS) **통합 인증**
  - **ISMS**: 인증기준 80개 (관리체계 16 + 보호대책 64)
  - **ISMS-P**: 인증기준 **101개** (ISMS 80개 + 개인정보 처리단계별 21개)
- **법적 근거**
  - 정보보호: **정보통신망법 제47조**
  - 개인정보: **개인정보보호법 제32조의2**
  - 통합고시: 「정보보호 및 개인정보보호관리체계 인증 등에 관한 고시」
- **운영 거버넌스**
  - 정책기관: 과학기술정보통신부 + 개인정보보호위원회
  - 인증기관: KISA, 금융보안원(FSI)
  - 심사기관: KISA, 차세대정보보안인증원, 한국정보통신진흥협회 등

### ISMS 의무대상 (암기 핵심 — 자주 출제)
다음 중 하나라도 해당하면 **의무**:
1. ISP (정보통신망서비스 제공자, 회선설비 보유)
2. IDC (집적정보통신시설 사업자)
3. 연간 매출 **100억 이상** 또는 **이용자 일평균 100만명 이상**(전년도 말 직전 3개월 평균)인 정보통신서비스 제공자
4. 매출 **1,500억 이상**인 의료법상 상급종합병원·학교(재학생 1만명 이상 대학)

> **함정**: ISMS-P(개인정보 포함)는 의무가 아니라 **자율 선택**. 의무대상은 어디까지나 ISMS.

## 1-2. 인증기준 101개 구조 (시험 최대 비중)

```
A. 관리체계 수립 및 운영 (16개)   ← PDCA 순서, 운영 중에 지속 수행
   1.1 관리체계 기반 마련 (6)
   1.2 위험 관리 (4)
   1.3 관리체계 운영 (3)
   1.4 관리체계 점검 및 개선 (3)

B. 보호대책 요구사항 (64개)        ← 12개 분야
   2.1 정책,조직,자산 관리 (3)
   2.2 인적 보안 (6)
   2.3 외부자 보안 (4)
   2.4 물리 보안 (7)
   2.5 인증 및 권한관리 (6)
   2.6 접근통제 (7)
   2.7 암호화 적용 (2)
   2.8 정보시스템 도입 및 개발 보안 (6)
   2.9 시스템 및 서비스 운영관리 (7)
   2.10 시스템 및 서비스 보안관리 (9)
   2.11 사고 예방 및 대응 (5)
   2.12 재해복구 (2)

C. 개인정보 처리단계별 요구사항 (21개)  ← 생명주기(LifeCycle) 순서
   3.1 수집 시 보호조치 (7)
   3.2 보유·이용 시 보호조치 (5)
   3.3 제공 시 보호조치 (4)
   3.4 파기 시 보호조치 (2)
   3.5 정보주체 권리보호 (3)
```

> **암기 팁**: 분야별 개수를 외워두면, 결함 사례를 어느 분야로 분류할지 빠르게 좁힐 수 있다.
> A는 **PDCA**(계획→운영→점검→개선) 흐름, C는 **수집→보유→제공→파기→권리** 흐름으로 기억.

### 핵심 기준 의미 (A. 관리체계)
- **1.1.1 경영진 참여** / **1.1.2 최고책임자 지정(CISO/CPO)** / **1.1.4 범위 설정** / **1.1.5 정책 수립**
- **1.2.1 정보자산 식별 → 1.2.2 흐름분석 → 1.2.3 위험평가 → 1.2.4 보호대책 선정** (위험관리 4단계, 순서 출제됨)
- **1.4.1 법적 요구사항 준수 검토 / 1.4.2 관리체계 점검(내부감사) / 1.4.3 개선**

# 2부. 핵심 암기사항 (숫자·기간·법령)

## 2-1. 제도 운영 기간 ⭐최빈출

| 항목 | 기간 |
|------|------|
| 의무대상 인증 취득기한 | **차년도 8월 31일까지** |
| 결함 **보완조치** | **40일** 이내 |
| 보완조치 **재조치**(연장) | 추가 **60일** (합 **100일**) |
| **이의신청** | 결과 통보 후 **15일** 이내 |
| 심사기관 **보완요구** | 심사 종료 다음날부터 **30일** 이내 |
| **사후심사** | **1년** 주기(매년) |
| **갱신심사** | **3년** 주기, 유효기간 만료 **3개월 전** 신청 |
| 인증 유효기간 | **3년** |
| 심사기관 지정 유효기간 | **3년** (연장 신청은 만료 **6개월 전**) |

> **헷갈림 주의**: 보완조치 **40일** vs 이의신청 **15일** vs 보완요구 **30일**. 숫자 섞이기 쉬움 → 표로 통암기.

## 2-2. 개인정보 관련 기간 ⭐최빈출

| 항목 | 기간 |
|------|------|
| 개인정보 **파기** | 보유목적 달성·불필요해진 때부터 **5일** 이내(지체 없이) |
| 정보주체 **열람·정정 등 회신** | **10일** 이내 |
| 개인정보 **유출 통지**(정보주체) | **72시간** 이내 |

| **접속기록 보관** | 최소 **1년** / (5만명 이상 민감·고유식별정보 또는 기간통신사업자) **2년** |
| 접속기록 **점검** | **월 1회 이상** |

## 2-3. 안전성 확보조치

**암호화 저장 필수 정보**: 고유식별정보, **비밀번호**(일방향 암호화), 바이오정보(생체인식정보), 주민등록번호, 신용카드번호, 계좌번호, 여권번호 등

**내부 관리계획 포함사항**(대표적 항목): 책임자 지정, 역할·책임, 교육계획, 접근권한 관리, 접근통제, 암호화, 접속기록 관리, 악성프로그램 방지, 물리적 안전조치, 위험분석, 유출 대응계획, 수탁자 관리 등

## 2-4. 암호 알고리즘 ⭐기술문제 단골

| 구분 | 안전 (사용 가능) | 미안전 (사용 금지) |
|------|------------------|-------------------|
| 대칭키 | **SEED, ARIA, AES, HIGHT, LEA** | DES, 3DES(점진 폐지), RC4 |
| 해시 | **SHA-256/384/512** | **SHA-1, MD5** |
| 공개키 | RSA(2048↑), ECC | RSA-1024 이하 |

> 국산 알고리즘 = SEED, ARIA, HIGHT, LEA (헷갈리면 "**세아하리**"로 묶어 암기)

## 2-5. 손해배상책임보험 (가입 의무 기준)

- 적용대상(✅공식확인, 개인정보보호법 시행령 §48의7): **저장·관리 정보주체 일일평균 1만명 이상** **AND** **전년도 매출액등 10억원 이상**
- 최저가입금액은 **정보주체 수 × 매출액 구간별 별표**로 정해짐(시행령 별표1의3). 구간이 세분화되어 있으므로, 구체적 금액(예: 대규모 처리자 10억원 등)은 **시행령 별표 원문으로 확인**할 것 — 암기보다 "1만명+10억 기준" 적용 여부가 핵심.

## 2-6. 수수료 감면

**정보보호 공시 기업 30% 할인**(✅KISA 공식 확인). 그 외 소기업·인증심사 일부생략 등 감면은 KISA 수수료 산정기준에 따름 — **구체 감면율(소기업 30%, 생략 20% 등)은 KISA 수수료 안내로 확인 필요**(공식 페이지에서 명시 미확인).

# 3부. 헷갈리는 내용 — 유사 기준 비교

수험생이 가장 많이 틀리는 "이거랑 저거 중 뭐로 결함 잡지?" 포인트.

### ① 2.5(인증·권한관리) vs 2.6(접근통제)
- **2.5** = "누가 들어오는가" — **계정/식별/인증/비밀번호/권한 부여·검토**
- **2.6** = "어디까지 닿는가" — **네트워크/시스템/응용/DB/무선/원격/인터넷 차단·통제**
- 예: 퇴직자 계정 미삭제 → 2.5.1(계정관리) or 2.2.5(퇴직관리). DB에 모든 직원이 접근 가능 → 2.6.4.

### ② 2.5.5(특수권한 관리) vs 2.5.6(접근권한 검토)
- 2.5.5 = 관리자 등 **특수계정 자체의 최소 부여·식별**
- 2.5.6 = 부여된 권한을 **주기적으로 재검토·회수**(검토 안 하면 2.5.6 결함)

### ③ 2.9.4(로그 관리) vs 2.9.5(로그 점검)
- 2.9.4 = 로그를 **남기고 보관**(미생성·보관기간 미달 결함)
- 2.9.5 = 남긴 로그를 **정기적으로 점검·분석**(점검 안 하면 2.9.5 결함)

### ④ 1.2.3(위험평가) vs 1.4.2(관리체계 점검)
- 1.2.3 = 구축 단계의 **위험 식별·평가**(연 1회 위험평가)
- 1.4.2 = 운영 단계의 **내부 감사/점검**(연 1회 사후관리 점검)

### ⑤ 3.1.1(수집·이용 동의) vs 3.3.1(제3자 제공 동의) vs 3.3.2(처리위탁)
- **3.1.1** = 우리가 직접 수집·이용 (동의 4항목 고지)
- **3.3.1** = **제3자에게 제공**(제공받는 자 이익 위해) → **별도 동의 필수**
- **3.3.2** = **위탁**(우리 업무 대신 처리, 예: 택배·콜센터) → **동의 불필요, 처리방침 공개로 충분**
- > 핵심 차이: **제공=상대방 목적 / 위탁=우리 목적**. 시험 최빈출 함정!

### ⑥ 3.1.4(민감정보·고유식별정보) vs 3.1.3(주민번호)
- 주민번호는 **법령 근거 없으면 동의받아도 처리 불가**(원칙적 금지, 3.1.3)
- 그 외 고유식별정보(여권·운전면허·외국인번호)·민감정보(건강·사상·노조 등)는 **별도 동의 시 처리 가능**(3.1.4)

### ⑦ 동의 vs 법적 근거 (수집 6대 적법 근거)
동의 / 법률 규정 / 계약 이행 / 정보주체 급박한 생명·신체·재산 이익 / 정당한 이익 / 공공기관 소관업무. → "**동의만이 적법근거다**"는 **오답**.

### ⑧ 가명정보(3.2.5) vs 익명정보
- 가명정보 = **추가정보 사용 시 재식별 가능** → 개인정보(통계·연구·공익기록 목적 동의 없이 처리 가능, 단 결합은 전문기관)
- 익명정보 = 재식별 **불가능** → 개인정보 아님(규제 대상 외)

# 3부-B. 보호대책·개인정보 분야별 빈출 결함 사례 (암기 핵심)

> 실제 시험의 70% 이상이 "이 사례는 어느 기준 결함?"이다. 분야별로 **전형적 결함 패턴**을 통째로 익혀두면 매핑 속도가 빨라진다.

### 2.4 물리 보안
- 2.4.1 서버실을 '보호구역'으로 지정하지 않음 / 등급 미구분
- 2.4.2 출입기록을 남기지 않거나 방문자 동행 없이 출입
- 2.4.6 USB·노트북 등 반출입 기기 통제대장 없음
- 2.4.7 **퇴근 시 책상에 서류 방치(Clean Desk 위반)**, 화면잠금 미설정

### 2.5 인증 및 권한관리
- 2.5.2 공용계정(admin, guest) 공유 사용 → **개인별 식별 불가**
- 2.5.3 중요시스템에 **단일 패스워드만**(MFA 미적용)
- 2.5.4 비밀번호 평문 저장, 복잡도·변경주기 미적용
- 2.5.5 **관리자 권한을 전 직원이 보유**, 특수권한 목록 미관리
- 2.5.6 권한 부여 후 **재검토를 한 번도 안 함**

### 2.6 접근통제
- 2.6.1 방화벽 정책에 **Any-Any 룰** 존재, 미사용 정책 방치
- 2.6.2 OS 접근 세션 타임아웃 미설정
- 2.6.4 **개발자가 운영 DB에 직접 접속** 가능
- 2.6.6 원격접근(VPN)에 MFA 없이 ID/PW만
- 2.6.7 업무망에서 인터넷 자유 접속(망분리 대상인데 미적용)

### 2.7 암호화
- 2.7.1 고유식별정보·비밀번호 **평문 저장/전송**
- 2.7.2 **암호키를 소스코드에 하드코딩**, 키 변경·폐기 절차 없음

### 2.8 개발 보안
- 2.8.3 **개발·운영 환경 미분리**(같은 서버)
- 2.8.4 **운영 DB의 실데이터를 테스트에 그대로 사용**(미변환)
- 2.8.5 소스코드 형상관리·접근통제 없음

### 2.9 운영관리
- 2.9.2 시스템 장애 대응절차·임계치 모니터링 없음
- 2.9.3 백업은 하지만 **복구 테스트를 안 함**
- 2.9.4 로그 보관기간 미달 / 2.9.5 로그 점검 미수행
- 2.9.7 폐기 PC·HDD를 **데이터 완전삭제 없이 매각**

### 2.10 서비스 보안관리
- 2.10.1 IPS/WAF **탐지 이벤트를 검토하지 않음**
- 2.10.2 클라우드 보안 책임(RACI)·설정 점검 미흡
- 2.10.8 **OS·SW 보안패치 장기간 미적용**
- 2.10.9 백신 미설치 단말, 패턴 자동 업데이트 안 됨

### 2.11 사고 대응
- 2.11.1 침해사고 대응절차·비상연락망 없음
- 2.11.2 **취약점 점검 후 조치를 안 함**(점검만 함)
- 2.11.4 모의훈련 미실시

### 3.1 개인정보 수집
- 3.1.1 동의 시 **4대 고지사항**(항목·목적·보유기간·거부권리) 누락
- 3.1.1 **필수/선택 동의를 한꺼번에** 받음(선택을 필수처럼)
- 3.1.2 서비스에 불필요한 과다 정보 수집
- 3.1.5 간접수집 후 정보주체 요구 시 **수집출처 미고지**
- 3.1.7 회원가입 동의에 **마케팅 동의를 끼워넣음**(별도 동의 X)

### 3.2 보유·이용
- 3.2.4 **수집목적과 다른 용도로** 개인정보 이용(목적 외)
- 3.2.5 가명처리 추가정보를 분리보관하지 않음

### 3.3 제공
- 3.3.1 제3자 제공 시 별도 동의·고지 누락
- 3.3.2 위탁 사실·수탁자를 **처리방침에 미공개**, 수탁사 교육·점검 안 함
- 3.3.4 국외이전 시 고지·동의·보호조치 누락

### 3.4 파기
- 3.4.1 보유기간 경과 정보를 **파기하지 않고 보관**
- 3.4.1 파기 시 **복구 가능한 방식**(단순 삭제)으로 처리

### 3.5 권리보호
- 3.5.1 처리방침을 **홈페이지 첫 화면에서 쉽게 찾을 수 없음**, 변경 이력 미관리
- 3.5.2 열람·정정·삭제 요구 절차(창구) 부재 또는 10일 초과

### [문제 1] 제도 기간 (단순질의형)
인증심사 결과 결함에 대한 **보완조치 기간**과, 부득이한 경우 연장(재조치)까지 포함한 **최대 기간**으로 옳은 것은?

① 30일 / 60일  ② 40일 / 100일  ③ 15일 / 45일  ④ 40일 / 60일  ⑤ 30일 / 90일

**정답: ②**
**해설**: 기본 보완조치 **40일**, 재조치(연장) **60일** 추가 → 합계 **100일**. ①의 30/60, ④의 40/60(합산 오해)이 대표 함정. 이의신청 15일과 혼동 주의.

### [문제 2] 제3자 제공 vs 위탁 (복합응용형)
다음 중 정보주체의 **별도 동의가 반드시 필요한** 경우는?

① 쇼핑몰이 배송을 위해 택배사에 주소·연락처를 전달
② 콜센터 업무를 외주업체에 위탁하며 고객 DB 접근 권한 부여
③ 보험사가 **제휴 카드사의 상품 판매**를 위해 고객정보를 카드사에 넘김
④ 클라우드(IaaS)에 개인정보 DB를 저장·운영
⑤ 그룹웨어 유지보수 업체가 점검 중 개인정보에 접근

**정답: ③**
**해설**: ①②④⑤는 모두 **위탁**(우리 업무를 대신 처리, 수탁자는 우리 목적을 위해 일함) → 처리방침 공개로 충분, **동의 불필요**(3.3.2). ③만 **제3자 제공**(카드사 자기 목적·이익을 위해 사용) → **별도 동의 필수**(3.3.1). 판별 기준은 "누구의 이익·목적이냐".

### [문제 3] 인증기준 매핑 (상황판단형)
"A사는 퇴직한 직원의 VPN 계정과 사내 포털 계정이 퇴직 **3개월 후에도** 삭제되지 않은 채 남아 있었다." 가장 직접적인 결함 기준은?

① 2.2.5 퇴직 및 직무변경 관리
② 2.5.1 사용자 계정 관리
③ 2.6.6 원격접근 통제
④ 2.5.6 접근권한 검토
⑤ 2.2.6 보안 위반 시 조치

**정답: ①** (2.5.1도 부분 연관)
**해설**: 핵심 원인은 "**퇴직 처리 시 접근권한 회수 절차 미흡**"이므로 **2.2.5**가 가장 근본(Root cause). 계정 관리 절차 자체 문제면 2.5.1, 권한을 주기적으로 재검토하지 않은 점을 강조하면 2.5.6. 시험은 "가장 적절한/근본적인" 기준을 요구하므로 **사건의 트리거(퇴직)**에 주목.

### [문제 4] 암호 알고리즘 (단순질의형)
개인정보 보호를 위해 **사용해서는 안 되는** 알고리즘만 묶은 것은?

① AES, SEED  ② SHA-256, ARIA  ③ **MD5, SHA-1, DES**  ④ LEA, HIGHT  ⑤ ARIA, AES

**정답: ③**
**해설**: MD5·SHA-1(취약 해시), DES·RC4(취약 대칭키)는 사용 금지. 안전 대칭키 = SEED/ARIA/AES/HIGHT/LEA, 안전 해시 = SHA-256 이상. 비밀번호는 **일방향(해시) 암호화** 저장이라는 점도 함께 출제됨.

② 정보주체 열람 요구 회신 — 10일 이내
③ 개인정보 유출 통지 — 72시간 이내
④ 접속기록 보관 — 최소 6개월
⑤ 접속기록 점검 — 월 1회 이상

### [문제 6] 위험관리 순서 (단순질의형)
ISMS-P "1.2 위험 관리"의 올바른 수행 순서는?

① 위험평가 → 자산식별 → 흐름분석 → 보호대책 선정
② **자산식별 → 흐름분석 → 위험평가 → 보호대책 선정**
③ 흐름분석 → 자산식별 → 보호대책 선정 → 위험평가
④ 자산식별 → 위험평가 → 흐름분석 → 보호대책 선정
⑤ 보호대책 선정 → 위험평가 → 자산식별 → 흐름분석

**정답: ②**
**해설**: 1.2.1 정보자산 식별 → 1.2.2 현황·흐름분석 → 1.2.3 위험평가 → 1.2.4 보호대책 선정. **무엇이 있는지(자산)** 알아야 **어떻게 흐르는지** 보고, 그 위에서 **위험을 평가**하고 **대책을 정한다**는 논리.

### [문제 7] 로그 관리 vs 점검 (상황판단형)
"B사는 서버 접속로그를 1년간 보관하고 있으나, 보관만 할 뿐 이상행위 여부를 **한 번도 분석하지 않았다**." 결함 기준은?

① 2.9.4 로그 및 접속기록 관리
② **2.9.5 로그 및 접속기록 점검**
③ 2.11.3 이상행위 분석 및 모니터링
④ 2.9.2 성능 및 장애관리
⑤ 2.10.1 보안시스템 운영

**정답: ②**
**해설**: 로그를 **남기고 보관**하는 건 2.9.4(여기선 충족). 문제는 "**점검·분석을 안 함**" → **2.9.5** 결함. (개인정보 접속기록은 월 1회 이상 점검 의무.) 2.11.3은 실시간 위협탐지 체계 관점이라 결이 다름.

② 집적정보통신시설(IDC) 사업자

### [문제 9] 가명정보 (상황판단형)
가명정보에 대한 설명으로 **옳지 않은** 것은?

① 추가정보를 사용하면 특정 개인을 다시 식별할 수 있다
② 통계작성, 과학적 연구, 공익적 기록보존 목적은 정보주체 동의 없이 처리 가능
③ **익명정보와 동일하게 개인정보보호법 적용을 받지 않는다**
④ 서로 다른 개인정보처리자 간 가명정보 결합은 지정된 전문기관을 통해야 한다
⑤ 가명처리에 사용한 추가정보는 분리 보관·관리해야 한다

**정답: ③**
**해설**: 가명정보는 **여전히 개인정보**이므로 보호법 적용 대상(안전조치·결합 제한 등). 재식별이 **불가능한 익명정보**만 법 적용 제외. 가장 흔한 함정 보기.

### [문제 10] 처리위탁 결함 (상황판단형)
"C사는 콜센터 업무를 D사에 위탁하면서, 위탁 사실을 **개인정보 처리방침에 공개하지 않았고** 수탁사 교육·점검도 하지 않았다." 가장 적절한 결함 기준은?

① 3.3.1 개인정보 제3자 제공
② **3.3.2 개인정보 처리 업무 위탁**
③ 3.2.4 목적 외 이용 및 제공
④ 2.3.2 외부자 계약 시 보안
⑤ 3.5.1 개인정보 처리방침 공개

**정답: ②**

# 5부. 마무리 학습 전략

1. **암기 표 3장**(제도기간 / 개인정보기간 / 암호알고리즘)은 시험 직전까지 반복 — 단순질의형 점수 보장.
2. **101개 기준은 "분야 → 의도" 단위로 이해**. 외우려 하지 말고 "이 결함은 누가/어디/언제의 문제냐"로 분류하는 훈련.
3. **헷갈림 7세트(3부)**를 사례로 구분 연습 — 복합·상황판단형 핵심.
